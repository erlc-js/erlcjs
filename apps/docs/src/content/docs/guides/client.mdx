The `Client` class is the central coordinator of erlc.js. It coordinates HTTP REST calls, handles token/key management, and manages the cache and event emitters.

## Initialization Options

To construct a new `Client` instance, pass in a config object conforming to the `ClientOptions` interface:

```typescript
import { Client } from '@erlcjs/core';

const client = new Client({
    serverKey: 'YOUR_SERVER_KEY',
    globalKey: 'OPTIONAL_GLOBAL_KEY',
    polling: true,
});
```

### Option Fields

| Field | Type | Description |
| :--- | :--- | :--- |
| `serverKey` | `string` | **Required.** The Server API key retrieved from your private server settings. |
| `globalKey` | `string` | *Optional.* A global key generated for your application. |
| `globalAppId` | `string | number` | **Optional.** The global app ID generated for your application. Used to create authorization links.
| `polling` | `boolean` | *Optional.* If `true`, the client requests server data updates every 5 seconds. |
| `webhook` | `object` | *Optional.* Configuration details for Webhook Gateway mode. |

---

## Connection Strategies

erlc.js offers two ways to receive real-time updates from your ER:LC private server: **Polling Mode** and **Webhook Gateway Mode**.

### 1. Polling Mode

In Polling Mode, the client triggers a background timer that fetches server logs and status every 5 seconds.
*   **Pros:** Requires no public IP, port forwarding, or domain names. Great for local developer environments or simple scripts.
*   **Cons:** Higher resource utilization (HTTP overhead) and up to a 5-second delay for game events.

```javascript
const client = new Client({
    serverKey: 'YOUR_KEY',
    polling: true // Starts polling immediately upon construction
});
```

:::danger
Webhooks are required to receive custom command events
:::

### 2. Webhook Gateway Mode

In Webhook Gateway Mode, your script spins up a local HTTP server that waits for incoming POST webhooks pushed by ER:LC server hooks.
*   **Pros:** Extremely fast (near real-time latency), highly efficient (no constant polling requests).
*   **Cons:** Your server must expose a public port (or use a tunnel like ngrok) to allow Roblox servers to reach it.

```javascript
const client = new Client({
    serverKey: 'YOUR_KEY',
    webhook: {
        enabled: true,
        port: 3000,
        path: '/webhook',
        secret: 'my-webhook-secret' // Verify requests security
    }
});
```

When Webhook Gateway mode is enabled, the client spawns an internal Node.js HTTP server listening on the configured port and path. You can subscribe to the same events as you would in polling mode.

:::danger
It is recommended to still have polling enabled to receive the full suite of events. This is because webhooks only receive emergency call and custom command events.
:::
